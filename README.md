# 应急物资轮换和借用系统

一个完整的应急物资管理系统，支持物资批次管理、借用审批流程、库存轮换、审计追踪等功能。

## 技术栈

- **后端**：Python 3.x + Flask + SQLAlchemy + SQLite
- **前端**：Vue 3 + Vite + Element Plus + Pinia
- **认证**：JWT Token
- **数据存储**：SQLite 本地数据库（无需额外安装）

## 快速启动

### 1. 启动后端服务

```bash
# 进入后端目录
cd backend

# 安装依赖（首次运行）
pip install -r requirements.txt

# 启动服务
python app.py
```

后端服务将在 `http://localhost:5000` 启动，首次启动会自动创建数据库和初始化用户账号。

### 2. 启动前端服务

```bash
# 进入前端目录
cd frontend

# 安装依赖（首次运行）
npm install

# 启动开发服务
npm run dev
```

前端服务将在 `http://localhost:5173` 启动。

### 3. 访问系统

打开浏览器访问 `http://localhost:5173`

## 数据模型

### 物资批次 (MaterialBatch)
- `batch_no`：批次号（唯一）
- `material_name`：物资名称
- `specification`：规格型号
- `total_quantity`：总数量
- `available_quantity`：可用数量
- `unit`：单位
- `production_date`：生产日期
- `expiry_date`：有效期至
- `warehouse_location`：库位
- `status`：状态（normal/rotated）
- `supplier`：供应商
- `remark`：备注

### 借用单 (BorrowOrder)
- `order_no`：单据号
- `status`：状态（pending_approval/approved/rejected/received/returned/damaged）
- `purpose`：借用用途
- 审批人、领用人、归还人及时间

### 审计日志 (AuditLog)
记录每一次状态变化，包含：
- 操作类型、操作人、操作时间
- 原因/备注（必填）
- 旧状态 → 新状态
- 操作详情
- 盘点关联：stock_take_id, old_quantity, new_quantity（库存调整时记录数量变化）

### 盘点单 (StockTake)
- `stock_take_no`：盘点单号（唯一，格式PD+日期+序号）
- `material_batch_id`：关联批次ID
- `status`：状态（pending_confirm/confirmed/cancelled）
- `expected_quantity`：账面数量
- `actual_quantity`：实盘数量
- `difference_quantity`：差异数量
- `difference_reason`：差异原因
- `handling_opinion`：处理意见
- 创建人、确认人、撤销人及时间

**步骤 2：发起借用申请**

**步骤 3：审批借用单**

**步骤 6：查看审计历史**
1. 进入「借用单据管理」→ 点击单据「详情」
2. 查看完整的状态流转记录和审计日志
3. 进入「操作审计日志」，查看全系统操作记录

### 链路二：异常场景验证（库存不足、提前归还、过期借用）

**场景 1：借用数量超过库存**
1. 确认某批次可用库存（如 90 盒）
2. 发起借用申请，选择该批次，填写数量 100
3. 系统提示：「批次 XXX 库存不足，可用 90盒，申请 100盒」
4. 申请被拒绝，库存保持不变

**场景 2：未领用就归还**
1. 找到一个状态为「待审批」或「待领用」的单据
2. 操作列中不会显示「归还」按钮
3. 直接调用 API 会返回：「当前状态为XXX，无法归还」
4. 库存保持不变

**场景 3：借用已过期批次**
1. admin 创建一个批次，设置有效期为过去的日期（如 2023-01-01）
2. duty 发起借用申请，在物资选择列表中看不到该过期批次
3. 查看该批次详情，状态显示「已过期」
4. 直接调用 API 会返回：「批次 XXX 已过期，不能借用」

**场景 4：报损处理**
1. 领用一批物资（状态变为「使用中」）
2. 点击「报损」，填写报损数量和原因
3. 确认后，单据状态更新，已报损数量增加
4. 物资库存不恢复（因为已报损）

## 业务规则总结

1. **库存变更时机**：仅在「领用出库」时扣减库存，「归还入库」时增加库存，「盘点确认」时调整库存
2. **状态流转控制**：
   - 借用单：待审批 → （审批通过/拒绝）→ 待领用/已拒绝 → （领用）→ 使用中 → （归还/报损）→ 已归还/已报损
   - 盘点单：待确认 → （确认/撤销）→ 已确认/已撤销
3. **备注强制要求**：所有状态变更操作必须填写原因/备注，盘点有差异时必须填写差异原因
4. **过期批次隔离**：已过期批次在借用选择列表中自动过滤
5. **并发安全**：每次操作都在事务中执行，失败自动回滚，保证库存一致性
6. **审计追踪**：所有操作都记录日志，可追溯到操作人、时间、原因，库存调整记录调整前后数量
7. **盘点权限控制**：
   - 仓库管理员：发起盘点、录入实盘、确认盘点、撤销盘点、查看所有盘点单（含待确认）
   - 值班员：仅能查看已完成的盘点单（已确认 + 已撤销两种终态），不能发起、录入、确认或撤销任何盘点单
8. **盘点冲突检测**：
   - 同一批次同一时间只能有一个未完成（待确认）的盘点单
   - 批次存在待审批/待领用/未完成借用单时，不能发起或确认盘点
9. **盘点库存调整**：确认盘点后，批次可用库存和总库存同时调整为实盘数量
10. **数据持久化**：盘点单、调整后的库存、审计日志均持久化到SQLite，重启服务后数据不丢失

## 常见问题

**Q: 后端启动报错缺少依赖？**
A: 确保已执行 `pip install -r requirements.txt`

**Q: 前端启动报错找不到模块？**
A: 确保已执行 `npm install`

**Q: 数据库想要重置？**
A: 删除 `backend/data.db` 文件，重新启动后端会自动初始化

**Q: 如何修改端口？**
A: 后端修改 `app.py` 最后一行的 port，前端修改 `vite.config.js` 中的 server.port
